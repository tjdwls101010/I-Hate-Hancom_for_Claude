# HWPX Format Technical Guide

Technical reference for the HWPX XML format. This covers format structure and XML construction rules — for design decisions, see `document-design.md`; for style IDs, see `style-catalog.md`.

---

## 1. Architecture: HWPX as a ZIP Package

An HWPX file is a ZIP archive with this structure:

```
document.hwpx (ZIP)
├── mimetype                    ← MUST be first entry, STORED (not compressed)
├── META-INF/
│   ├── container.xml
│   ├── container.rdf
│   └── manifest.xml
├── Contents/
│   ├── header.xml              ← Style definitions (never hand-write)
│   ├── content.hpf             ← Package manifest
│   └── section0.xml            ← Document content (this is what you write)
├── Preview/
│   ├── PrvText.txt
│   └── PrvImage.png
├── settings.xml
└── version.xml
```

**You only write `section0.xml`.** Everything else comes from the base template or is generated by `build_hwpx.py`.

## 2. The ID Reference System

HWPX separates style definitions from content, similar to CSS/HTML:

- **header.xml** defines styles as numbered IDs:
  - `charPr` (character properties): font, size, bold, color
  - `paraPr` (paragraph properties): alignment, indentation, line spacing
  - `borderFill`: border styles and background fills for table cells
  - `fontface`: font name definitions referenced by charPr

- **section0.xml** references those IDs:
  - `<hp:p paraPrIDRef="0">` → uses paragraph style #0
  - `<hp:run charPrIDRef="9">` → uses character style #9
  - `<hp:tc borderFillIDRef="22">` → uses border/fill style #22

Every ID you reference MUST exist in header.xml. The validator checks this.

## 3. XML Namespaces

section0.xml uses these namespace prefixes:

```xml
<hs:sec xmlns:hs="http://www.hancom.co.kr/hwpml/2011/section"
  xmlns:hp="http://www.hancom.co.kr/hwpml/2011/paragraph"
  xmlns:hp10="http://www.hancom.co.kr/hwpml/2016/paragraph"
  xmlns:hc="http://www.hancom.co.kr/hwpml/2011/core"
  xmlns:hh="http://www.hancom.co.kr/hwpml/2011/head"
  xmlns:hwpunitchar="http://www.hancom.co.kr/hwpml/2016/HwpUnitChar">
```

The root element is `<hs:sec>`. All paragraphs, runs, and tables use the `hp:` prefix.

## 4. Section XML Structure

### Paragraphs

Every visible element is inside a paragraph (`<hp:p>`):

```xml
<hp:p id="0" paraPrIDRef="0" styleIDRef="0" pageBreak="0" columnBreak="0" merged="0">
  <hp:run charPrIDRef="9">
    <hp:t>Visible text content here</hp:t>
  </hp:run>
</hp:p>
```

Required attributes on `<hp:p>`: `id`, `paraPrIDRef`, `styleIDRef`, `pageBreak`, `columnBreak`, `merged`.

- Set `pageBreak="1"` to force a page break before this paragraph.
- `styleIDRef="0"` is always safe (default style).
- `id="0"` is always safe.

### Runs

A paragraph contains one or more runs (`<hp:run>`). Each run has a single character style:

```xml
<hp:p ...>
  <hp:run charPrIDRef="9"><hp:t>Normal text </hp:t></hp:run>
  <hp:run charPrIDRef="34"><hp:t>bold keyword</hp:t></hp:run>
  <hp:run charPrIDRef="9"><hp:t> more normal text</hp:t></hp:run>
</hp:p>
```

Split text into separate runs whenever the character style changes.

### secPr (Page Setup)

The first paragraph's first run must contain `<hp:secPr>` — the page setup block (margins, dimensions, footnote config). **Do not write this yourself.** `build_hwpx.py` auto-injects it from `templates/base/secpr_template.xml` if your section0.xml doesn't include one.

## 5. Table Construction

Tables are the trickiest part of HWPX. Follow this structure exactly.

### Basic Table Structure

```xml
<!-- Table lives INSIDE a run, inside a paragraph -->
<hp:p id="0" paraPrIDRef="0" styleIDRef="0" pageBreak="0" columnBreak="0" merged="0">
  <hp:run charPrIDRef="0">
    <hp:tbl rowCnt="2" colCnt="3" cellSpacing="0" borderFillIDRef="4"
            pageBreak="CELL" repeatHeader="0" id="0">
      <!-- Row 1 -->
      <hp:tr>
        <hp:tc ...>...</hp:tc>  <!-- Cell (0,0) -->
        <hp:tc ...>...</hp:tc>  <!-- Cell (1,0) -->
        <hp:tc ...>...</hp:tc>  <!-- Cell (2,0) -->
      </hp:tr>
      <!-- Row 2 -->
      <hp:tr>
        <hp:tc ...>...</hp:tc>  <!-- Cell (0,1) -->
        <hp:tc ...>...</hp:tc>  <!-- Cell (1,1) -->
        <hp:tc ...>...</hp:tc>  <!-- Cell (2,1) -->
      </hp:tr>
    </hp:tbl>
  </hp:run>
</hp:p>
```

**Critical rules:**
- `rowCnt` must equal the actual number of `<hp:tr>` elements
- `colCnt` must equal the number of columns (accounting for colSpan)
- Tables go directly inside `<hp:run>`, NEVER inside `<hp:ctrl>`
- `borderFillIDRef` on `<hp:tbl>` is the table's outer border (usually "4")

### Cell Structure

Every `<hp:tc>` must have these attributes and children in this exact order:

```xml
<hp:tc name="" header="0" hasMargin="0" protect="0" editable="0" dirty="0" borderFillIDRef="4">
  <hp:cellAddr colAddr="0" rowAddr="0"/>
  <hp:cellSpan colSpan="1" rowSpan="1"/>
  <hp:cellSz width="16000" height="800"/>
  <hp:cellMargin left="283" right="283" top="141" bottom="141"/>
  <hp:subList>
    <hp:p paraPrIDRef="22" styleIDRef="0">
      <hp:run charPrIDRef="0"><hp:t>Cell content</hp:t></hp:run>
    </hp:p>
  </hp:subList>
</hp:tc>
```

**cellAddr rules:**
- `colAddr` starts at 0 and increments by 1 for each cell in a row
- If a cell has `colSpan="2"`, the next cell's `colAddr` jumps by 2
- `rowAddr` is the row index (0-based)

**cellSz:**
- `width` is in HWPUNIT (1 mm ≈ 283 HWPUNIT, 1 cm ≈ 2835)
- Sum of all cell widths in a row should equal total table width
- Content area width: page width (59528) - left margin (5669) - right margin (5669) = **48190**. Use ~48000 for full-width tables.

**cellMargin:**
- Standard values: `left="283" right="283" top="141" bottom="141"` (1mm horizontal, 0.5mm vertical)

### Simplified Cell in subList

Inside `<hp:subList>`, paragraphs use a simplified form — no `id`, `columnBreak`, `merged`:

```xml
<hp:p paraPrIDRef="22" styleIDRef="0">
  <hp:run charPrIDRef="136"><hp:t>Header text</hp:t></hp:run>
</hp:p>
```

### ColSpan Example

To merge cells horizontally:

```xml
<!-- 3-column table where first row has a merged cell spanning cols 1-2 -->
<hp:tr>
  <hp:tc ... borderFillIDRef="22">
    <hp:cellAddr colAddr="0" rowAddr="0"/>
    <hp:cellSpan colSpan="1" rowSpan="1"/>
    <hp:cellSz width="10000" height="800"/>
    ...
  </hp:tc>
  <hp:tc ... borderFillIDRef="22">
    <hp:cellAddr colAddr="1" rowAddr="0"/>
    <hp:cellSpan colSpan="2" rowSpan="1"/>  <!-- spans 2 columns -->
    <hp:cellSz width="38000" height="800"/> <!-- combined width -->
    ...
  </hp:tc>
  <!-- No third tc needed — colSpan="2" covers it -->
</hp:tr>
```

## 6. Units and Measurements

| Unit | Meaning | Conversion |
|------|---------|------------|
| HWPUNIT | Base unit for dimensions | 1 mm = 283.46 HU |
| Font height | charPr `height` attribute | value / 100 = point size (e.g., 1400 = 14pt) |
| Spacing | Line spacing `value` | 160 = 160% of line height |

**Common widths:**
- A4 page: 59528 HU (210mm)
- Left/right margin: 5669 HU each (20mm)
- Content area: ~48190 HU (170mm)
- Standard cell margin: 283 HU horizontal, 141 HU vertical

## 7. Hard Constraints

These cause Hancom Office to crash, show blank pages, or corrupt the document:

| # | Constraint | Consequence of Violation |
|---|-----------|------------------------|
| 1 | mimetype must be first ZIP entry, STORED | File won't open |
| 2 | All charPrIDRef, paraPrIDRef, borderFillIDRef must exist in header.xml | Crash or render failure |
| 3 | Table rowCnt/colCnt must match actual rows/columns | Crash |
| 4 | cellAddr must be sequential per row | Cell misalignment or crash |
| 5 | secPr must be complete (all child elements) | Blank pages |
| 6 | Tables must be inside `<hp:run>`, not `<hp:ctrl>` | Crash |
| 7 | `<hp:tc>` must have all 7 attributes | Render failure |
| 8 | XML encoding must be UTF-8 | Encoding errors |
| 9 | Namespace prefixes must be consistent | Parse failure |

All of constraints 1 and 5 are handled automatically by `build_hwpx.py`. You only need to worry about 2-4 and 6-9 when writing section0.xml.

## 8. Quick Reference: Minimal Valid section0.xml

```xml
<?xml version="1.0" encoding="UTF-8" standalone="yes" ?>
<hs:sec xmlns:hs="http://www.hancom.co.kr/hwpml/2011/section"
  xmlns:hp="http://www.hancom.co.kr/hwpml/2011/paragraph"
  xmlns:hp10="http://www.hancom.co.kr/hwpml/2016/paragraph"
  xmlns:hc="http://www.hancom.co.kr/hwpml/2011/core"
  xmlns:hh="http://www.hancom.co.kr/hwpml/2011/head"
  xmlns:hwpunitchar="http://www.hancom.co.kr/hwpml/2016/HwpUnitChar">

  <hp:p id="0" paraPrIDRef="19" styleIDRef="0" pageBreak="0" columnBreak="0" merged="0">
    <hp:run charPrIDRef="128">
      <hp:t>문서 제목</hp:t>
    </hp:run>
  </hp:p>

  <hp:p id="0" paraPrIDRef="0" styleIDRef="0" pageBreak="0" columnBreak="0" merged="0">
    <hp:run charPrIDRef="9">
      <hp:t>　　본문 텍스트입니다.</hp:t>
    </hp:run>
  </hp:p>

</hs:sec>
```

This is the simplest valid document. `build_hwpx.py` will inject secPr and package it. For realistic documents, copy patterns from the design catalog.
